# Register Distillation

*Turns verified units into an output register. Adds nothing.*

You are producing a register — a scholarly apparatus, a sermon, an audio script,
a simplified explainer, slides, a translation-only edition, or similar — named in
the project. Read `PROJECT.md` and `CONVENTIONS.md` first.

## The one rule

A register may **compress, reorder, translate, omit, and reframe for its
audience.** A register may **never add propositional content** — no fact,
quotation, ruling, or attribution that is not already in a **verified** unit.

## Rules

1. Draw only from units whose status is `verified`. If you need something that
   lives only in an unverified or non-existent unit, stop and request that unit
   be composed and verified first.
2. Maintain a **claim-to-source map**: every substantive claim in the register
   points back to the unit (and span) it derives from. Save it alongside the
   register so review can target only what is new — which should be nothing.
3. Match the register to its audience's register of language, but never let
   accessibility introduce a claim the core does not support. Simplify the
   expression, not the evidence.
4. High-trust-by-audience outputs (sermons, audio narration) must draw from
   `verified-exact` / `verified-translation` units only — never `paraphrase` or
   `unverified`.
5. Write the register to `registers/` (project-level) or
   `units/NNNN-*/registers/` (unit-level), per the project's layout.

If you cannot express something faithfully in the target register, say so and
leave it out rather than approximate it.
